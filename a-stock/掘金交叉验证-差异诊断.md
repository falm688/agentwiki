# 掘金量化交叉验证 — 差异诊断

## 对比摘要

| 指标 | SQL引擎 | 掘金量化 | 差异 |
|:----|:------:|:--------:|:----:|
| 总收益 | +470.52% | **+13.79%** | -456pp |
| 年化 | +31.28% | ~+2.6% | -28.7pp |
| 最大回撤 | 19.1% | **-55.01%** | +35.9pp |
| 胜率 | ~58% | 50.7% | -7.3pp |
| 交易次数 | ~220 | 151 | -69笔 |

## 已发现的三大问题

### 问题1：股票池只包含沪市股票（严重）

**现象**：交易记录中 **0只深市股票**，全部46只股票均来自SHSE。

**可能原因**：
- `get_instruments()` 在掘金量化上的返回格式可能不同（symbol字段是否含交易所前缀？）
- 也可能是深市股票的symbol格式为 `000001` 而非 `SZSE.000001`，导致前缀过滤异常

**影响**：候选池腰斩。V4策略在更大池子中更容易找到符合条件的标的。

### 问题2：持仓卡死 — `safe_price` 失败后跳过卖出检查（严重BUG）

**现象**：`605338(巴比食品)` 2021-05-06买入后**4.5年未卖出**。+25天理应到期卖出（HD=25），却一直持有到回测结束。

**原因**：代码中存在静默跳过BUG
```python
for sym in list(context.v4_pos.keys()):
    px = safe_price(sym, context, 1)
    if px is None or len(px) == 0:
        continue   # ← BUG：取价失败时跳过卖出，股票永远卡住
```

当 `history_n` 因任何原因（停牌、数据缺失、订阅问题）返回None时，卖出逻辑被**静默跳过**，该股票永远留在 `v4_pos` 中，再也无法卖出。后续每天同样跳过，形成死循环。

**影响**：导致多只股票永久套牢。603929（+¥248万）、603983（+¥356万）、605338（+¥275万）均在持仓中一直未卖出。

### 问题3：气候分类器 CASH 模式锁死（2021.12-2022.10、2023.09-12）

**现象**：**15个月零开仓**。具体：
- 2021-12 ~ 2022-10：11个月无交易
- 2023-09 ~ 2023-12：4个月无交易

**原因**：气候分类器 CASH 条件全年触发：
```python
if vol20 > VOL20_LOW and above_ma60 < 0.5 and ret60 < -0.08:
    cash_days += 1   # 连续计数
effective_cash = cash_days >= 3  # 3天确认
```

2022年HS300从4900跌到3500（-28%），`ret60`长期低于-0.08，`above_ma60=0`。更严重的是 —— 即使市场开始反弹，`ret60`仍需**60个交易日**才能恢复正常。意味着CASH模式会在底部区域**额外锁死2-3个月**。

**矛盾**：V4策略的设计本意是在下跌中买入（60日跌>8%），但气候分类器却在大跌时阻止买入。

## 次要问题

### HD使用日历日而非交易日

掘金量化的 `(datetime - buy_date).days` 计算**自然日**，25自然日 ≈ 18个交易日。而SQL引擎使用交易日。导致持仓时间更短，错过部分涨幅。

### 气候分类器V3.1与最终SQL引擎版本是否一致？

需要检查 `bt_tianshi_final.py` 中使用的分类器版本是否与掘金脚本完全一致。V3.1还包含反弹检测（bounce=0.02），但看起来没有生效。

## 结论

**SQL引擎结果不可信** —— 掘金量化是独立框架，跑出来的结果差这么多，说明：
1. SQL引擎可能有计算错误（价格匹配、交易执行、回撤计算等）
2. 气候分类器设计本身有缺陷（下跌时锁仓矛盾）
3. 需要修复掘金脚本的BUG后重新跑，才能得到可信的对比数据

**建议下一步**：先修复脚本再对比，否则无法判断是SQL引擎还是掘金的问题。
