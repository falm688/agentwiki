# 量化交易系统架构 — 最终方案v2

> **一句话**：各策略独立模拟盘运行（各自资金/持仓），不设全局仲裁层。核心难点是打板的虚拟成交逻辑和回测引擎的日内多时间窗支持。

## 二、各策略独立模拟盘

每个策略独立：
- 初始资金: ¥100,000
- 状态文件: `sim_{strategy_id}_state.json`
- 每日信号: `sim_{strategy_id}_daily.json`
- 执行时间: 波段14:50 / 打板09:25+14:50

状态文件格式统一：

## 三、打板虚拟成交方案

这是最核心的技术难题。模拟盘在09:25要知道"这个买单能不能成交"。

### 方案：规则驱动 + 事后校准

```
09:25 竞价数据到达
  │
  ├── 候选股满足条件（开幅/量比/占比） → 产生买入Signal
  │
  ├── 成交判定（规则驱动）：
  │    ├── 竞价涨幅 < 5%  → ✅ 大概率成交（买入价=竞价价）
  │    ├── 竞价涨幅 5-7% → ⚠️ 可能排队（标记为"拟成交"）
  │    └── 竞价涨幅 > 7% → ❌ 大概率买不到（放弃）
  │
  ├── "拟成交"状态 → 14:50用当日实际数据校准
  │    ├── 当日该股确实涨停 → 确认成交
  │    └── 当日没涨停/炸板 → 撤销（未成交）
  │
  └── 记录决策日志：为什么成交/为什么放弃
```

### 为什么用规则驱动而不是概率模型
- 规则可理解、可调整
- 不需要大量历史数据训练
- 打板策略本身胜率68.9%，成交率偏差的影响可控

## 四、回测引擎 — 支持日内多时间窗

```python
# 回测推进机制
for date in date_range:
    for strategy in strategies:
        if strategy.type == '波段':
            data = loader.load_daily(date)
            signals = strategy.generate_signals(date, data)
        elif strategy.type == '打板':
            # 09:25时间窗 — 只能用竞价前数据
            auction_data = loader.load_auction(date)
            signals_am = strategy.generate_timeframe_signals(date, auction_data, 'open_call_auction')
            
            # 14:50时间窗 — 只能用14:50前数据
            close_data = loader.load_close_cutoff(date)  # 截断到14:50
            signals_pm = strategy.generate_timeframe_signals(date, close_data, 'close')
```

每个时间窗的数据**严格截断**，不回测未来。
